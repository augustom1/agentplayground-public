AgentPlayground Documentation

AgentPlayground is a self-hosted, multi-tenant AI agent management platform. Users create and manage Agent Teams — named groups of Claude-powered agents — assign tasks, schedule recurring jobs, chat with Claude through a tool-use interface that reads and writes the database, and build a growing catalog of reusable skills.

For agents reading this: This documentation is structured for both human readers and automated agents searching for how to interact with the platform. Every endpoint, permission scope, chat tool name, and data model is listed below.

Architecture

The system follows a Next.js App Router structure with three main layers:

  • UI Layer — React 19 pages inside app/(app)/, protected by NextAuth middleware.
  • API Layer — REST API routes under app/api/. All routes validate the JWT session, check the user's role, then query Prisma.
  • Data Layer — Prisma 7 ORM over PostgreSQL with pgvector. A singleton client in lib/prisma.ts.

The chat endpoint streams Claude responses. Claude uses tool definitions from lib/chat-tools.ts, which call permission-gated functions in lib/db-agent.ts. This is the primary intelligence interface — Claude can read/write any model it has permission for.

Local Development

1. Clone the repo

git clone https://github.com/agentplayground/agent-dashboard-ui-app.git
cd agent-dashboard-ui-app

2. Create .env

DATABASE_URL="postgresql://postgres:postgres@localhost:5432/agentdb"
AUTH_SECRET="replace-with-32-char-random-string"
ANTHROPIC_API_KEY="sk-ant-api03-..."
NEXTAUTH_URL="http://localhost:3000"
CRON_SECRET="replace-with-random-secret"

3. Start Postgres and Redis

docker compose up -d

4. Install and migrate

npm install
npx prisma migrate dev

5. Run the dev server

npm run dev
# App runs at http://localhost:3000

First Run — Create Admin Account

On first startup, navigate to /setup. This page is only accessible when no users exist in the database. It calls POST /api/auth/setup to create the first admin user and runs lib/seed-defaults.ts to seed default skills and widget configuration.

After setup, you are redirected to /login. Log in with the credentials you just created.

The /setup route is automatically blocked by middleware once the first user exists.

Run this command to populate the database with five pre-built agent teams:

VPS Deployment

For a full production deployment on a Linux VPS (Ubuntu 22+):

DNS setup

Add two A records pointing to your VPS IP: @ → VPS IP and * → VPS IP. This covers the root domain and all subdomains.

Configure .env.local

DOMAIN=yourdomain.com
ANTHROPIC_API_KEY=sk-ant-...
AUTH_SECRET=your-32-char-secret
POSTGRES_USER=dbuser
POSTGRES_PASSWORD=strongpassword
POSTGRES_DB=agentdb
REDIS_PASSWORD=redispass
CRON_SECRET=cron-secret
NEXTAUTH_URL=https://app.yourdomain.com

Run setup script

bash setup.sh

The script installs Docker, starts all services with docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d --build, waits for the health check at /api/health, and pulls Ollama models.

Dashboard

Route: /dashboard. The main landing page after login. Fetches data from GET /api/metrics and GET /api/widgets.

Displays configurable widgets:

  • Stat widgets — total teams, agents, tasks, users.
  • Feed widget — recent activity log (latest 10 entries across all teams).
  • Calendar widget — upcoming scheduled jobs for the next 7 days.
  • Custom/Optimization widget — skill count, CLI function count, and improvement log stats.
  • Team widget cards (components/TeamWidget.tsx) — per-team status badge, agent count, recent task activity.

The metrics API aggregates counts across all DB models. The dashboard auto-refreshes via components/RefreshButton.tsx.

Chat

Route: /chat. Streams Claude responses via POST /api/chat. Uses the Anthropic SDK's streaming API (stream method).

Conversations are persisted. Each conversation has a title and a list of messages with role: user | assistant. Managed via /api/conversations.

Claude operates with a base system prompt that describes the AgentPlayground mission and available tools. It is instructed to prefer automation over manual steps, log improvements, and generate tools when it detects patterns.

See Chat Tool Definitions for the full list of tools Claude can call.

Agent Lab

Route: /agent-lab. A sandboxed playground for testing prompts against any agent team.

  • Select a target team from the dropdown.
  • Enter a prompt and run it.
  • Results are displayed in the run output panel.
  • Run history is stored in PlaygroundRun records (/api/playground-runs).

Schedule

Route: /schedule. Calendar view of all upcoming and past jobs.

Two types of scheduled work:

  • Scheduled Jobs (ScheduledJob model) — one-off or recurring events (daily/weekly/monthly). Created via POST /api/schedule.
  • Recurring Tasks (RecurringTask model) — cron-expression-based tasks with timezone support. Created via POST /api/recurring-tasks. Executed by POST /api/cron.

Tools Catalog

Route: /tools. Aggregates the platform's internal tool library per team.

Three categories:

  • Skills — Named, categorized instruction sets Claude uses when performing tasks for a team.
  • CLI Functions — Registered shell commands or scripts. Flagged as dangerous if they have destructive side effects.
  • Improvements — Logged optimizations detected by Claude. Each has a category, an impact level, and an applied flag.

The aggregation endpoint is GET /api/tools, which returns all three categories merged with team metadata and counts.

Claude can add to this catalog via the log_improvement and generate_tool chat tools (see Chat Tool Definitions).

Users (Admin Only)

Route: /users. Only accessible to users with role: admin. The middleware blocks all non-admin access and returns a 403.

Admin can view, create, update roles, and deactivate user accounts via /api/users.

Settings

Route: /settings. User-level settings including display preferences and account management. Writes to the User model.

Auth & Roles

Authentication is handled by NextAuth v5 using the JWT strategy — no database hit per request. Passwords are hashed with bcrypt.

The JWT session payload includes:

{
  id: string,
  email: string,
  role: "admin" | "user" | "viewer",
  plan: "free" | "pro" | "enterprise"
}

Middleware in middleware.ts protects all routes. Public routes: /login, /setup, /api/auth/*.

The /users and /api/users routes are additionally gated to admin role only.

Agent Permission Presets

Defined in lib/agent-permissions.ts. Each team is assigned a permission preset which determines what the db-agent layer allows Claude to do on behalf of that team.

PresetCapabilities
admin Full read + write across all models, CLI execution, team management, user reads
builder Read all, write own team's data, file read/write, CLI access, create teams
standard Read + write own team's tasks and agents, file read only
readonly Read own team data only — no writes

Chat Tool Definitions

All tools available to Claude during chat sessions are defined in lib/chat-tools.ts and call lib/db-agent.ts with permission gating.

Tool nameDescription
get_teamsList all agent teams with status, agent count, and recent activity
get_team_detailsFull details for a specific team including all agents
get_tasksFetch tasks optionally filtered by teamId and status
create_taskCreate a new task assigned to a team
update_task_statusUpdate a task's status (pending → running → completed | failed)
get_metricsRetrieve dashboard metrics: agent counts, task stats, optimization data
get_skillsList skills for a team or globally
create_skillCreate a new skill for a team
get_improvementsList logged improvements, filterable by applied/unapplied
log_improvementLog a detected optimization pattern with category, description, and impact
generate_toolConvert a repeated workflow into a permanent skill or CLI function
get_scheduleList scheduled jobs and recurring tasks
create_schedule_eventCreate a new scheduled job
get_conversationsList chat conversations
get_activity_logFetch recent activity log entries
The self-improvement flywheel: Claude detects repeated patterns via log_improvement, then uses generate_tool to convert them into reusable skills stored in the catalog. Future tasks route to the tool instead of calling the LLM.

Cron Jobs

Recurring tasks are executed by calling POST /api/cron from an external scheduler (n8n, crontab, or any HTTP client). 

curl -X POST https://app.yourdomain.com/api/cron \
  -H "Authorization: Bearer $CRON_SECRET"

The endpoint picks up all RecurringTask records where enabled = true and nextRunAt <= now(), executes them, and updates lastRunAt and nextRunAt.

Health check endpoint (used by Docker and setup.sh):

GET /api/health  →  { status: "ok", db: "connected" }

Local LLM (Ollama)

The stack includes a custom Ollama Docker image (Dockerfile.ollama) that auto-pulls models on first start via ollama-entrypoint.sh.

Default models pulled:

OLLAMA_AUTO_PULL="qwen2.5:3b qwen2.5:7b"

Override OLLAMA_AUTO_PULL in your environment to pull different models. Models are stored in a named Docker volume so they survive container restarts.

Open WebUI is available at https://ai.yourdomain.com and connects to Ollama internally. Model routing (local vs Claude) is manual — use team system prompts to instruct agents on which tasks to handle locally.
